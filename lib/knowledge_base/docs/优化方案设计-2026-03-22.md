## 优化1: 减少LLM调用 - Beat合并处理

### 1.1 现状分析

**代码位置**: `director_agent.py` 第250-298行 `orchestrate_scenes()`

**问题描述**:
```python
for beat in plot_outline.beats:
    if not beat.expected_chars:
        continue
    # 每个beat都触发一次 sub_agent_pool.batch_act()
    beat_responses = await sub_agent_pool.batch_act(...)
```

当章节有N个beat时，会产生N次LLM调用。如果beat数量多（如10+），LLM调用成本很高。

### 1.2 方案设计

#### 方案A: Beat分组合并 (推荐)
- **思路**: 将多个beat合并为一批，一次LLM调用处理多个beat
- **优点**: 显著减少调用次数，保留上下文连贯性
- **缺点**: 需要确保合并后的prompt不超过token限制

```python
# 伪代码
BEAT_BATCH_SIZE = 3  # 每3个beat合并处理

async def orchestrate_scenes_batched(self, plot_outline, cast, sub_agent_pool):
    # 将beats分组
    batches = [
        plot_outline.beats[i:i + BEAT_BATCH_SIZE]
        for i in range(0, len(plot_outline.beats), BEAT_BATCH_SIZE)
    ]

    for batch in batches:
        # 合并batch内的所有beat为一个场景描述
        combined_desc = self._combine_beats_description(batch)
        # 一次调用处理多个beat
        responses = await sub_agent_pool.batch_act(...)
```

#### 方案B: 并行+汇总模式
- **思路**: 先并行处理所有beat，最后LLM汇总
- **优点**: 最快速度，最少LLM调用
- **缺点**: 失去beat间的上下文关联

#### 方案C: 动态阈值触发
- **思路**: beat数量 > 阈值(默认5)才触发合并
- **优点**: 小章节保持原样，大章节自动优化
- **缺点**: 逻辑复杂度增加

### 1.3 实现路径

**关键代码改动点**:

```python
# director_agent.py 新增配置和合并方法

@dataclass
class DirectorAgentConfig:
    # ... 现有配置 ...
    beat_batch_threshold: int = 5  # 超过此数量自动合并
    beat_batch_size: int = 3      # 合并的批次大小

    async def orchestrate_scenes(
        self,
        plot_outline: PlotOutline,
        cast: List[CharacterBible],
        sub_agent_pool: Any,
    ) -> Dict[str, CharacterResponse]:
        """编排场景演绎 - 优化版支持beat合并"""
        if len(plot_outline.beats) > self.config.beat_batch_threshold:
            return await self._orchestrate_scenes_batched(
                plot_outline, cast, sub_agent_pool
            )
        return await self._orchestrate_scenes_original(
            plot_outline, cast, sub_agent_pool
        )

    async def _orchestrate_scenes_batched(
        self,
        plot_outline: PlotOutline,
        cast: List[CharacterBible],
        sub_agent_pool: Any,
    ) -> Dict[str, CharacterResponse]:
        """合并模式: 多个beat一次处理"""
        all_responses = {}
        batch_size = self.config.beat_batch_size

        for i in range(0, len(plot_outline.beats), batch_size):
            batch = plot_outline.beats[i:i + batch_size]

            # 构建合并的beat描述
            combined_desc = self._build_combined_beat_prompt(batch, plot_outline)

            # 收集本批次需要的所有角色
            batch_chars = set()
            for beat in batch:
                batch_chars.update(beat.expected_chars)

            # 一次LLM调用处理本批次所有角色
            beat_responses = await sub_agent_pool.batch_act(
                scene_description=combined_desc,
                characters=list(batch_chars),
            )

            # 分配响应到各个beat
            for char_name, response in beat_responses.items():
                if char_name not in all_responses:
                    all_responses[char_name] = []
                all_responses[char_name].append(response)

        return all_responses

    def _build_combined_beat_prompt(
        self,
        beats: List[PlotBeat],
        plot_outline: PlotOutline,
    ) -> str:
        """构建合并后的beat提示"""
        beat_sections = []
        for idx, beat in enumerate(beats):
            section = f"""=== 节拍 {idx + 1}: {beat.beat_type} ===

{beat.description}

【预期角色】: {', '.join(beat.expected_chars)}"""
            beat_sections.append(section)

        return f"""【场景设定】
{plot_outline.scene_setting}

【关键事件】
{', '.join(plot_outline.key_events)}

【悬念】
{plot_outline.suspense}

【本批次节拍】
{chr(10).join(beat_sections)}

请依次完成以上所有节拍的演绎，保持情节连贯性。"""
```

### 1.4 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 合并后单次prompt过长 | 中 | 设置batch_size上限(3-5)，监控token使用 |
| Beat上下文丢失 | 低 | 合并时传递完整场景设定，保持连贯性 |
| 角色响应归属混淆 | 低 | 每个响应标记beat类型，便于后续整合 |

## 优化2: 上下文截断 - ChapterMemory压缩

### 2.1 现状分析

**代码位置**: `data_structures.py` 第315-388行 `ChapterMemory`

**问题描述**:
```python
@dataclass
class ChapterMemory:
    chapter: int
    character_states: Dict[str, Dict[str, Any]]       # 角色当前状态
    character_state_history: List[CharacterStateChange]  # 状态变化历史
    relationship_states: Dict[str, Dict[str, str]]    # 关系状态
    relationship_history: List[RelationshipChange]       # 关系变化历史
    key_items: Dict[str, List[str]]                   # 关键物品
    item_changes: List[ItemChange]                     # 物品变化历史
    suspense_hooks: List[SuspenseHook]                 # 悬念钩子
    key_events_happened: List[str]                     # 关键事件
    verification_history: List[Dict[str, Any]]         # 验证历史
```

随着章节数增加，每个ChapterMemory都累积上述数据。如果有100章，每章都保留完整历史，上下文会膨胀。

### 2.2 方案设计

#### 方案A: 增量快照压缩 (推荐)
- **思路**: 每N章做一次"快照压缩"，只保留最近状态，清空历史
- **优点**: 实现简单，效果可控
- **缺点**: 历史信息完全丢失

```python
class ChapterMemory:
    # 新增字段
    _is_compressed: bool = False

    def compress(self, keep_recent_chapters: int = 3) -> "ChapterMemory":
        """压缩历史，保留最近N章的状态快照"""
        # 只保留最新的状态，丢弃历史列表
        return ChapterMemory(
            chapter=self.chapter,
            character_states=self.character_states.copy(),
            character_state_history=[],  # 清空
            relationship_states=self.relationship_states.copy(),
            relationship_history=[],      # 清空
            key_items=self.key_items.copy(),
            item_changes=[],             # 清空
            suspense_hooks=self.suspense_hooks.copy(),  # 保留未解决的悬念
            key_events_happened=[],      # 清空
            verification_history=[],      # 清空
        )
```

#### 方案B: 层级记忆系统
- **思路**: 区分"工作记忆"和"长期记忆"
  - 工作记忆: 最近3章完整保留
  - 长期记忆: 摘要压缩
- **优点**: 保留关键信息同时控制大小
- **缺点**: 实现复杂度高

#### 方案C: 摘要替换
- **思路**: 历史记录只保留"摘要"，而非原始数据
- **优点**: 信息密度高
- **缺点**: 需要LLM参与摘要生成

```python
# data_structures.py - ChapterMemory 新增方法

@dataclass
class ChapterMemory:
    # ... 现有字段 ...

    def compress(self, keep_recent_chapters: int = 3) -> "ChapterMemory":
        """压缩历史记录，保留最近N章的状态

        Args:
            keep_recent_chapters: 保留最近N章的完整记录

        Returns:
            压缩后的ChapterMemory
        """
        # 过滤出最近的变化记录
        recent_state_changes = [
            h for h in self.character_state_history
            if h.chapter >= self.chapter - keep_recent_chapters
        ]
        recent_relation_changes = [
            h for h in self.relationship_history
            if h.chapter >= self.chapter - keep_recent_chapters
        ]
        recent_item_changes = [
            c for c in self.item_changes
            if c.chapter >= self.chapter - keep_recent_chapters
        ]

        return ChapterMemory(
            chapter=self.chapter,
            character_states=self.character_states.copy(),
            character_state_history=recent_state_changes,
            relationship_states=self.relationship_states.copy(),
            relationship_history=recent_relation_changes,
            key_items=self.key_items.copy(),
            item_changes=recent_item_changes,
            suspense_hooks=self.suspense_hooks.copy(),  # 悬念全部保留
            key_events_happened=[],  # 清空，由下一章大纲重新定义
            verification_history=[],  # 清空
        )

    def get_context_summary(self) -> Dict[str, Any]:
        """生成上下文摘要，用于传递给LLM

        Returns:
            摘要字典，包含关键状态信息
        """
        return {
            "chapter": self.chapter,
            "character_states": self.character_states,
            "active_relationships": {
                k: v for k, v in self.relationship_states.items()
                if "仇" in v or "敌" in v or "爱" in v  # 只保留重要关系
            },
            "pending_suspense": [
                {"type": h.hook_type, "desc": h.description}
                for h in self.suspense_hooks if not h.is_resolved
            ],
            "recent_changes": {
                "characters": len(self.character_state_history),
                "relations": len(self.relationship_history),
                "items": len(self.item_changes),
            }
        }
```

### 2.4 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 重要历史信息丢失 | 中 | 保留最近3章完整历史，悬念钩子全部保留 |
| 跨章状态追踪困难 | 低 | 使用character_states快照而非完全清空 |
| 压缩时机不当 | 低 | 每5章压缩，提供配置项可调 |

## 优化3: 缓存优化 - SimpleCache增强

### 3.1 现状分析

**代码位置**: `cache.py` 第186-203行 `get_llm_response_cache()`

**问题描述**:
```python
_llm_response_cache = SimpleCache(CacheConfig(
    enabled=True,
    max_size=500,      # 默认500条
    ttl_seconds=1800,   # 30分钟
))
```

用户反馈512条可能不够用，且没有持久化。

### 3.2 方案设计

#### 方案A: 增大容量 + LRU调优 (推荐)
- **思路**: 增大默认容量，优化LRU淘汰逻辑
- **优点**: 改动小，效果明显
- **缺点**: 仍是内存缓存，有上限

```python
_llm_response_cache = SimpleCache(CacheConfig(
    enabled=True,
    max_size=2000,      # 2000条，适合中长篇
    ttl_seconds=3600,   # 1小时
    eviction_policy="lru",
))
```

#### 方案B: 分层缓存
- **思路**: L1(内存) + L2(磁盘) 分层
- **优点**: 容量大，支持持久化
- **缺点**: 实现复杂度增加

#### 方案C: 智能缓存
- **思路**: 根据cache key特征自动调整TTL
- **优点**: 热门内容更长尾
- **缺点**: 需要额外分析逻辑

### 3.3 实现路径

**关键代码改动点**:

```python
# cache.py - SimpleCache 增强

class SimpleCache:
    def __init__(self, config: Optional[CacheConfig] = None):
        self.config = config or CacheConfig()
        self._cache: Dict[str, CacheEntry] = {}
        self._lock = threading.RLock()
        self._hits = 0
        self._misses = 0
        # 新增: 记录缓存键的访问频率
        self._access_freq: Dict[str, int] = {}  # key -> hit count

    def set(self, key: str, value: Any):
        """设置缓存值，支持高频键延长TTL"""
        with self._lock:
            # 检查容量，触发淘汰
            if self.config.enabled and len(self._cache) >= self.config.max_size:
                self._evict()

            # 如果key已存在，更新但保留访问频率
            if key in self._cache:
                self._cache[key].value = value
                self._cache[key].created_at = time.time()
                return

            # 新键初始化
            # 高频键获得更长TTL
            entry = CacheEntry(
                key, value,
                ttl=self._compute_adaptive_ttl(key)
            )
            self._cache[key] = entry
            self._access_freq[key] = 0

    def _compute_adaptive_ttl(self, key: str) -> int:
        """根据访问频率动态调整TTL"""
        base_ttl = self.config.ttl_seconds
        freq = self._access_freq.get(key, 0)

        # 访问超过10次的键，TTL翻倍
        if freq > 10:
            return base_ttl * 2
        # 访问超过50次的键，TTL三倍
        elif freq > 50:
            return base_ttl * 3
        return base_ttl

    def _evict(self):
        """LRU淘汰 + 低频键优先淘汰"""
        if not self._cache:
            return

        # 计算每个键的优先级: 低频 + 久未访问 = 高优先级被淘汰
        def eviction_score(k: str) -> tuple:
            entry = self._cache[k]
            freq = self._access_freq.get(k, 0)
            last_access = entry.last_accessed
            # 分数越低越应该淘汰
            return (freq * 0.3 + (time.time() - last_access) * 0.7, k)

        # 淘汰分数最低的5%或至少1个
        items_to_evict = max(1, int(len(self._cache) * 0.05))
        sorted_keys = sorted(self._cache.keys(), key=eviction_score)

        for i in range(items_to_evict):
            evicted_key = sorted_keys[i]
            del self._cache[evicted_key]
            if evicted_key in self._access_freq:
                del self._access_freq[evicted_key]
            logger.debug(f"Evicted entry: {evicted_key[:8]}...")


def get_llm_response_cache() -> SimpleCache:
    """获取LLM响应缓存 - 增大容量"""
    global _llm_response_cache
    if _llm_response_cache is None:
        _llm_response_cache = SimpleCache(CacheConfig(
            enabled=True,
            max_size=2000,      # 2000条 (原500)
            ttl_seconds=3600,   # 1小时 (原30分钟)
            eviction_policy="lru",
        ))
    return _llm_response_cache
```

### 3.4 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 内存占用增加 | 低 | 可配置max_size，默认2000仍属合理范围 |
| 缓存过期不一致 | 低 | 自适应TTL基于访问频率，热点内容延长 |
| 线程安全问题 | 低 | 使用RLock保证线程安全 |

## 优化4: 验证提前 - Early Verification

### 4.1 现状分析

**代码位置**: `novel_orchestrator.py` 第139-285行 `generate()`

**问题描述**:
当前流程:
```
1. Director.create_plot_outline
2. Director.determine_cast
3. Director.orchestrate_scenes
4. Director.assemble_plot
5. NovelWriter.write_chapter  <-- 昂贵的LLM调用
6. OutlineEnforcer.enforce    <-- 验证失败则5浪费
```

验证在NovelWriter之后，如果验证失败，NovelWriter的调用就浪费了。

### 4.2 方案设计

#### 方案A: Assemble后立即验证 (推荐)
- **思路**: 在assemble_plot之后、novelize之前增加early verification
- **优点**: 提前发现大纲问题，避免novelize浪费
- **缺点**: 需要将enforcer引入orchestrator

```python
# 优化后的流程
1. Director.create_plot_outline
2. Director.determine_cast
3. Director.orchestrate_scenes
4. Director.assemble_plot
5. [NEW] Early Verification - 验证情节骨架
   - 如果失败: 返回feedback，Director重新编排
6. NovelWriter.write_chapter
7. Final Verification (可选)
```

#### 方案B: 两阶段验证
- **思路**: assemble后做轻量验证，novelize后做完整验证
- **优点**: 双重保障
- **缺点**: 调用两次enforcer

#### 方案C: 并行预验证
- **思路**: 在后台预验证多个可能的assemble结果
- **优点**: 最快发现错误
- **缺点**: 实现复杂，需要预测性生成

```python
# novel_orchestrator.py - 新增方法

def _early_verify_assembled_plot(
    self,
    assembled: AssembledPlot,
    chapter_context: ChapterContext,
    enforcer: OutlineEnforcer,
) -> VerificationReport:
    """Early verification - 在novelize前验证情节骨架

    这是一个轻量级验证，只检查:
    1. 关键事件是否在narration中体现
    2. 角色时刻是否覆盖必要角色
    3. 悬念是否埋设
    """
    from .outline_enforcer import CoverageReport, VerificationIssue

    outline = chapter_context.outline
    issues = []

    # 1. 检查关键事件覆盖率 (基于narration文本)
    events_found = []
    events_missing = []

    for event in outline.key_events:
        if event.lower() in assembled.narration.lower():
            events_found.append(event)
        else:
            events_missing.append(event)

    event_coverage = len(events_found) / len(outline.key_events) if outline.key_events else 1.0

    if events_missing:
        issues.append(VerificationIssue(
            issue_type="MISSING_EVENTS",
            details=events_missing,
            severity="HIGH",
            suggestion=f"情节骨架中缺少关键事件: {', '.join(events_missing)}",
        ))

    # 2. 检查角色覆盖
    required_chars = set(outline.metadata.get('character_appearances', []))
    mentioned_chars = set(assembled.character_moments.keys())
    uncovered_chars = required_chars - mentioned_chars

    if uncovered_chars:
        issues.append(VerificationIssue(
            issue_type="MISSING_CHARACTERS",
            details=list(uncovered_chars),
            severity="HIGH",
            suggestion=f"情节骨架中缺少角色: {', '.join(uncovered_chars)}",
        ))

    # 3. 检查悬念埋设
    if not assembled.scene_transitions and not assembled.narration[-200:]:
        issues.append(VerificationIssue(
            issue_type="SUSPENSE_MISSING",
            details=["章节结尾缺少悬念"],
            severity="MEDIUM",
            suggestion="在scene_transitions中添加悬念",
        ))

    # 计算评分
    score = 10.0
    if events_missing:
        score -= 2.0 * (len(events_missing) / len(outline.key_events))
    if uncovered_chars:
        score -= 1.0

    passed = score >= enforcer.PASS_THRESHOLD and not any(
        i.severity == "HIGH" for i in issues
    )

    return VerificationReport(
        chapter=chapter_context.chapter,
        passed=passed,
        score=round(score, 1),
        issues=issues,
        coverage=CoverageReport(
            event_coverage=event_coverage,
            events_found=events_found,
            events_missing=events_missing,
        ),
        timestamp=datetime.now().isoformat(),
    )
```

### 4.4 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Early verification漏检 | 低 | 保留final verification作为兜底 |
| 额外LLM调用开销 | 中 | Early verification是轻量规则检查，不调用LLM |
| 重试逻辑复杂化 | 中 | 使用feedback机制，让Director感知失败原因 |
